## Context

当前启动流程在 `main.go` 中集中管理：DB 初始化 → CA 检查 → 配置加载 → 代理启动 → HTTP 服务启动。任务只能通过 API 或代理拦截逐个创建，缺乏批量导入手段。

## Goals / Non-Goals

**Goals:**
- 启动时自动从 `task.txt` 批量导入任务
- 按行解析，奇数为名称、偶数为 URL，每两行为一个任务
- 根据 URL 后缀自动判定 Type
- 导入后删除 `task.txt`，避免重复导入

**Non-Goals:**
- 不支持在文件名或格式上做配置（固定为 `task.txt`）
- 不支持运行时热加载
- 不提供 API 接口

## Decisions

1. **在 `main.go` 中内联实现**，不新增 service 文件——仅涉及文件读取和解析，逻辑不超过 30 行，抽象为独立函数反而增加跳转成本。

2. **Type 判定规则**：URL 以 `.mp4` 结尾 → `video.MP4`，其余一律视为 `video.M3U8`。与代理拦截器 `GetVideo` 的判断方式一致。

3. **文件不存在则静默跳过**——无 `task.txt` 是常态，不应视为错误。仅在有文件但解析异常时打印日志。

4. **任务插入失败不阻断后续**——某一行数据异常不应影响其余任务的导入，日志记录并继续。

5. **导入完成后删除文件**——避免下次启动重复插入。若删除失败（如权限不足）仅输出 warning，不影响程序运行。

## Risks / Trade-offs

- [文件残留] 导入中途崩溃可能导致文件未被删除 → 下次启动会尝试重复插入，此时依赖 `doTask` 去重逻辑（按 URL 查重）兜底，已有任务的不会重复创建。
- [编码问题] `task.txt` 若非 UTF-8 编码可能导致名称乱码 → 不做处理，约定文件为 UTF-8。
